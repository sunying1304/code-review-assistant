# 代码评审标准说明

## 严重程度分级

| 等级 | 标识 | 含义 | 必须修复 |
|------|------|------|----------|
| Critical | 🔴 | 严重问题，影响安全性或导致程序崩溃 | 是 |
| Warning | 🟡 | 中等问题，影响性能、可维护性或存在潜在风险 | 建议 |
| Info | 🔵 | 轻微问题，代码风格或可选优化 | 可选 |

---

## 四个评审维度

### 1. 可读性（Readability）

评估代码是否易于理解和维护。

**Critical 级别**
- 完全无法理解的逻辑，缺少任何注释

**Warning 级别**
- 函数/变量命名不规范（不符合语言约定，如 Python 的 snake_case）
- 类名不符合 PascalCase 规范
- 函数过长（超过 50 行）
- 嵌套层级过深（超过 3 层）

**Info 级别**
- 缺少文档字符串（docstring）
- 参数名过于简短（单字母命名）
- 魔法数字未定义为常量
- 注释与代码不同步

---

### 2. 性能（Performance）

评估代码的时间/空间复杂度和资源使用效率。

**Critical 级别**
- 在主线程中进行同步阻塞 I/O（如同步 XHR）
- 无限循环或明显的死锁风险

**Warning 级别**
- O(n²) 或更高复杂度的算法（有更优解时）
- 在循环内重复执行昂贵操作（如重复查询数据库）
- 不必要的全量数据加载
- 大量字符串拼接（应使用 join 或模板字符串）

**Info 级别**
- 可以使用列表推导式替代 for 循环
- 可缓存的重复计算
- 不必要的类型转换

---

### 3. 安全性（Security）

评估代码是否存在安全漏洞。

**Critical 级别**
- SQL 注入：直接拼接用户输入到 SQL 语句
- XSS：直接将用户输入赋值给 innerHTML
- 硬编码密钥/密码/Token（出现在代码中）
- 命令注入：将用户输入传入 os.system/exec 等

**Warning 级别**
- 使用不安全的哈希算法（MD5/SHA1）存储密码
- 缺少输入验证和边界检查
- 非 timing-safe 的密码比较（直接使用 == 比较哈希）
- 敏感信息打印到日志

**Info 级别**
- 缺少 HTTPS 强制跳转
- Cookie 未设置 HttpOnly/Secure 标志
- 错误信息过于详细（可能泄露系统信息）

---

### 4. 最佳实践（Best Practice）

评估代码是否遵循语言和行业规范。

**Critical 级别**
- 裸 `except:` 或 `catch(e){}` 吞掉所有异常
- 修改函数参数中的可变默认值（Python 中的陷阱）

**Warning 级别**
- 直接暴露内部数据结构（应返回副本）
- 使用已废弃的 API
- 缺少必要的错误处理
- 全局可变状态滥用
- JavaScript 中使用 `var` 声明（应用 `const`/`let`）
- 使用 `==` 而非 `===`（JavaScript）

**Info 级别**
- 缺少单元测试覆盖
- 日志级别使用不当
- 函数职责不单一（Single Responsibility Principle）
- 重复代码（DRY 原则违反）

---

## 评分规则

| 分数区间 | 等级 | 说明 |
|----------|------|------|
| 90-100 | 优秀 | 几乎无问题，可直接合入 |
| 75-89 | 良好 | 有少量 Info/Warning，建议修复后合入 |
| 60-74 | 中等 | 有较多 Warning，需要修复主要问题 |
| 40-59 | 较差 | 有 Critical 或大量 Warning，必须修复 |
| 0-39 | 极差 | 存在严重安全漏洞，禁止合入 |

**扣分规则**：
- 每个 Critical：-15 分
- 每个 Warning：-5 分
- 每个 Info：-1 分
- 基础分：100 分
